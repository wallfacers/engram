# Feature Specification: 生物启发检索涨点 —— 顺序可测四枪闭合 Mem0 gap

**Feature Branch**: `003-bio-retrieval-locomo`

**Created**: 2026-07-19

**Status**: Draft

**Input**: User description: "生物启发检索涨点：顺序可测四枪闭合 Mem0 gap，在 LoCoMo 与 LongMemEval 上推到 85+。测量地基（多跑+95%CI+配对diff+LongMemEval adapter）为前置；四枪=模型regime对齐、实体平表→联想图攻multi-hop、检索侧时间结构化、拒答校准+冲突消解。local-first约束内，不碰BEAM。"

## Clarifications

### Session 2026-07-19

- Q: 涨点机制是否必须在与 Mem0 可比的检索预算内（single-pass、上下文 token 量级相当），还是只看准确率？ → A: 设预算门禁——single-pass（无 agentic 循环）+ 答题上下文 token 与现基线同量级（top_k=30 档），每枪报告 token 用量，涨分不得靠预算膨胀。
- Q: 冲突消解中被「压过」的旧记忆是物理删除，还是保留但降权/标记？ → A: 保留但压制——旧记忆标记「被取代」并指向取代者，默认检索降权/过滤，时间类查询与审计仍可见，误判可回退，不做物理删除。
- Q: 抽取模型在四枪期间是否冻结？ → A: 校准基线阶段一次性 A/B 对比两个抽取模型（各建库、各多跑），选定后全程冻结；此后所有配对比较基于同一记忆库，抽取贡献单独归因。（用户委托本方决定）
- Q: LongMemEval 用哪个口径？ → A: LongMemEval_S 全量（~500 题，业界主流报分口径）；M 版为非目标。附加要求：每次评测运行前 MUST 给出花费预估（预算账），运行后 MUST 记录实际花费。

## User Scenarios & Testing *(mandatory)*

本特性的「用户」是运行评测、调优记忆检索质量的维护者/研究者。核心诉求：在 local-first
约束内，把 engram 的长期对话记忆问答质量推到能与 Mem0 New Memory Algorithm
（2026-04：LoCoMo 92.5 / LongMemEval 94.4）扳手腕的量级，且**每一步涨点都可信地测量在
单跑噪声（±3-5pp）之上**，达不到就诚实回退。

五条 user story 按优先级排列，每条独立可测、独立交付价值。前两条（测量地基 + 模型校准
baseline）是后三条涨点机制成立的前提——没有可信基线，任何「涨点」都判不出真假。

### User Story 1 - 可信的多跑评测地基 (Priority: P1)

维护者需要一条命令，对完整数据集做多次重复评测，得到 per-category 与 OVERALL 的
**均值 ± 95% 置信区间**，以及同题集下两配置的**逐题配对差异报告**（含翻转题数与显著性）。
这样任何一次检索改动，都能判断它是真涨点，还是落在单跑噪声带内。

**Why this priority**: 该领域大量 <5pp「改进」实际在单跑噪声以内；答案生成随机性主导噪声
（两跑间约 59.5% 预测文本不同）。没有多跑 + 配对 diff，「一击必中」无从判断是否命中。
即使只交付这一条，也立刻带来价值——首次得到一个带误差棒的真实基线。

**Independent Test**: 对同一配置连续跑 N 次，验证输出包含每类别与总体的均值和 95% CI；
对两个已知有差异的配置跑配对 diff，验证报告正确列出翻转题目与配对显著性判定。

**Acceptance Scenarios**:

1. **Given** 一个完整评测数据集与答题端点, **When** 维护者以「重复 N 次」运行评测,
   **Then** 系统输出 per-category 与 OVERALL 的均值及 95% 置信区间。
2. **Given** 两个待比较的检索配置与固定题集, **When** 维护者运行配对差异比较,
   **Then** 系统报告逐题结果差异、翻转题数，并标注该差异是否高于噪声带（配对显著）。
3. **Given** LongMemEval 数据集, **When** 维护者以同一评测链运行,
   **Then** 系统按其题型正确加载、答题、判分并汇总，口径与 LoCoMo 评测一致可比。

---

### User Story 2 - 模型 regime 校准基线 (Priority: P1)

维护者需要在切换到强答题模型后重跑基线，得到「真架构 gap」——即分离「换前沿模型带来的
增益」与「记忆架构本身的贡献」。

**Why this priority**: 对标分数跑在强模型栈上，而现基线跑在较弱模型上；已知换前沿模型可带来
可观增益。不先校准，后续三枪的涨点会被「其实是模型涨的」污染，归因失真。

**Independent Test**: 在仅切换答题模型、其余配置不变的前提下运行多跑评测，得到带 CI 的
校准基线；确认该基线被记录为后续所有对比的参照点。

**Acceptance Scenarios**:

1. **Given** 一个更强的答题模型端点, **When** 维护者仅切换答题模型重跑多跑基线,
   **Then** 系统产出带 95% CI 的校准基线，且与旧基线的差异被明确记录为「模型贡献」。
2. **Given** 两个候选抽取模型, **When** 维护者在校准阶段分别建库并多跑对比,
   **Then** 系统报告两库的分差与 token 成本，选定者被记录为冻结抽取配置，后续机制评测
   不再更换。

---

### User Story 3 - 多跳联想检索 (Priority: P2)

维护者需要检索能沿实体关联跨条目遍历，取回多跳问题所需的互补事实，而非仅靠平表精确匹配。

**Why this priority**: multi-hop 是当前最清晰的弱项（约 66%），也是与图化竞品的结构性差距；
是架构层最高 ROI 的单点。

**Independent Test**: 在 multi-hop 类别上，用 P1 的多跑 + 配对 diff 对比开启/关闭联想检索，
验证 multi-hop 均值提升高于噪声带；对无需多跳的类别验证无显著回退。

**Acceptance Scenarios**:

1. **Given** 一组需要跨两个及以上条目才能回答的问题, **When** 启用联想检索,
   **Then** multi-hop 类别准确率均值相对校准基线提升且高于 95% CI 噪声带。
2. **Given** 单跳与其他类别题集, **When** 启用联想检索,
   **Then** 这些类别不出现高于噪声带的回退（联想检索不得引入 topic drift 净损害）。

---

### User Story 4 - 检索侧时间结构化 (Priority: P3)

维护者需要带时间意图的问题（"最近""上个月""去年"）在检索侧按时间范围结构化过滤后再匹配，
而不是仅把时间标记渲染给答题模型自行判断。

**Why this priority**: temporal 类别靠答题 prompt 难以救动，正确的层在检索侧。但需先证明其
提升真实——小样本上 temporal 已表现较好，须多跑验证不是撞噪声。

**Independent Test**: 在 temporal 类别上做多跑 + 配对 diff 对比开启/关闭时间结构化检索，
仅当提升高于噪声带才保留，否则记录为负结果并回退。

**Acceptance Scenarios**:

1. **Given** 带相对/绝对时间意图的问题集, **When** 启用检索侧时间结构化,
   **Then** temporal 类别准确率均值提升且高于噪声带；若未达标则该机制被回退并记录。

---

### User Story 5 - 拒答校准与冲突消解 (Priority: P3)

维护者需要系统在信息不足或问题带误导前提时**校准地拒答并指出缺了什么**，而非自信编造；
同时在写入侧对矛盾/包含的记忆做冲突消解，避免过时信息污染检索。

**Why this priority**: 对抗题约 27% 编造率直接拖累可比口径分（对抗题按「拒答为正确」记分）；
现有的重试式反 IDK 机制救可答题却害对抗题。

**Independent Test**: 在对抗题口径下多跑对比，验证编造率显著下降；同时验证可答题的误拒率
未显著上升（不对称：误拒可答题的代价更高）。

**Acceptance Scenarios**:

1. **Given** 一组带误导前提、答案不在记忆中的对抗题, **When** 启用校准拒答,
   **Then** 编造率相对基线显著下降，且系统在拒答时指出缺失信息。
2. **Given** 可答题集, **When** 启用校准拒答, **Then** 可答题误拒率不出现高于噪声带的上升。
3. **Given** 对同一事实存在矛盾或包含关系的多条记忆, **When** 触发冲突消解,
   **Then** 矛盾时新信息压过旧信息、包含时合并，过时信息不再污染检索结果。

### Edge Cases

- 当答题端点存在生成随机性时，单次结果不可作为涨点结论——必须经多跑 CI/配对 diff 判定。
- 当某机制在其目标类别提升落在噪声带内时，视为未达标，回退并如实记录（负结果亦为证据）。
- 当时间衰减类机制在单用户持久库上可能误删后续仍会被考的事实时，不做破坏性剪枝，
  仅做防抖，并以消融验证是否真涨再决定是否启用。
- 当联想检索引入话题漂移时，须以对原始问题的重排纠偏，且不得损害非多跳类别。
- 当语料规模超出 local-first 目标（远超约 10 万条记忆）时，本特性不承诺其表现（见 Assumptions）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持对完整评测数据集做「重复 N 次」评测，并输出 per-category 与
  OVERALL 的均值与 95% 置信区间。
- **FR-002**: 系统 MUST 支持对固定题集下两个配置的逐题配对差异比较，报告翻转题数与该差异
  是否高于单跑噪声带（配对显著性）。
- **FR-003**: 系统 MUST 支持以同一答题/判分/记分链在 LongMemEval 上评测，口径与 LoCoMo 可比；
  评测口径为 LongMemEval_S 全量（约 500 题），M 变体为非目标。
- **FR-004**: 系统 MUST 支持仅切换答题模型而不改其余配置，产出可作为后续对比参照的校准基线。
  校准阶段 MUST 一次性对比候选抽取模型（各自建库并多跑），选定后抽取模型在后续全部机制
  评测期间冻结不变，保证配对比较基于同一记忆库。
- **FR-005**: 系统 MUST 支持基于记忆间实体关联的多跳检索，使多跳问题能取回跨条目的互补事实。
- **FR-006**: 多跳检索 MUST NOT 在非多跳类别上引入高于噪声带的准确率回退。
- **FR-007**: 系统 MUST 支持在检索侧按时间范围对带时间意图的问题做结构化过滤，"最近/上个月"
  等相对表达以对话时间戳为锚归一化。
- **FR-008**: 系统 MUST 支持校准拒答：信息不足或前提被误导时拒答并指出缺失信息，而非编造。
- **FR-009**: 拒答机制 MUST 对「误拒可答题」施加高于「漏拒对抗题」的代价（不对称惩罚）。
- **FR-010**: 系统 MUST 在写入/整理侧对矛盾或包含关系的记忆做冲突消解，避免过时信息污染检索。
  压制 MUST 为非破坏性：被取代的旧记忆保留并标记「被取代」且指向取代者，默认检索
  降权/过滤，但时间类查询与审计路径仍可访问，误判可回退；MUST NOT 因冲突消解物理删除记忆。
- **FR-011**: 每一项涨点机制 MUST 可独立开启/关闭并独立测量，未达噪声带门槛者 MUST 可回退且
  其结果被如实记录。
- **FR-012**: 所有新增机制 MUST 在 local-first 约束内实现——不引入外部服务依赖、不引入
  会破坏无 CGO / 单二进制 / 单文件本地存储交付形态的依赖。
- **FR-013**: 所有涨点机制 MUST 在与对标可比的检索预算内运行：答题为 single-pass
  （无 agentic 循环），答题上下文 token 量与现基线同量级（top_k=30 档）；评测报告 MUST
  含每配置的 token 用量，任何涨分 MUST NOT 依赖检索预算膨胀取得。
- **FR-014**: 每次评测运行 MUST 先给出花费预估（按调用次数 × 模型单价的预算账），
  运行结束后 MUST 记录实际花费（token 用量与金额），预估与实际一并留档，供维护者控预算。

### Key Entities *(include if feature involves data)*

- **实体关联图**：记忆条目所含实体之间的关联（共现、同义），用于多跳联想遍历；属性含关联强度。
- **时间范围**：记忆事件的起止时间与相对时间锚点，用于检索侧时间过滤与时间相关性打分。
- **记忆冲突关系**：两条记忆间的兼容/矛盾/包含关系分类，驱动冲突消解与信息新鲜度维护；
  含「被取代」标记与指向取代者的引用（非破坏性压制，可审计、可回退）。
- **评测重复结果集**：同配置多次运行的逐题结果集合，用于计算均值、置信区间与配对差异。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: LoCoMo OVERALL 准确率均值（含对抗题按拒答记分的可比口径）在多跑评测下达到
  ≥ 85，且其 95% 置信区间下界不低于校准基线。
- **SC-002**: LongMemEval OVERALL 准确率均值在多跑评测下达到 ≥ 85。
- **SC-003**: multi-hop 类别准确率均值从约 66 提升到 ≥ 80，且提升高于 95% 噪声带。
- **SC-004**: 对抗题编造率从约 27% 显著下降（相对降幅经多跑验证高于噪声带），且可答题误拒率
  相对基线的上升不高于噪声带。
- **SC-005**: 每一项被保留（未回退）的涨点机制，其目标类别的提升均可复现地高于单跑 ±3-5pp
  噪声带（以 95% CI 不重叠或配对显著为判据）。
- **SC-006**: 全部机制在 local-first 交付形态下可运行——单二进制、无外部服务、本地单文件存储，
  规模覆盖至约 10 万条记忆量级。
- **SC-007**: 最终保留配置的答题上下文 token 用量与校准基线同量级（不超过其 1.5 倍），
  且全程保持 single-pass 答题（无 agentic 检索循环）。

## Assumptions

- 测量由维护者运行并控制预算；重复次数 N 与目标模型端点由维护者按预算设定（本特性提供能力，
  不固定 N 的具体取值）。
- 校准基线使用维护者提供的更强答题模型端点（如 gpt-5.6-sol），经由既有模型端点配置通道接入。
- LongMemEval 数据集（LongMemEval_S 变体）可由维护者提供，题型可映射到既有答题/判分/记分链；
  其抽取/建库开销显著（每题独立史料），由维护者分批控预算执行。
- 目标规模为 local-first 单用户约 10 万条记忆量级；千万 token 级语料（BEAM 1M/10M）为非目标，
  不承诺其表现，亦不在本轮替换本地检索引擎。
- 时间衰减/遗忘类机制在单用户持久库上默认不做破坏性剪枝，仅做防抖，是否启用衰减以消融验证为准。
- 跨论文/跨产品的绝对分不可直接比较；本特性只借鉴机制并以各来源内部相对幅度为参照，
  自身结论一律以本仓库统一评测口径下的多跑结果为准。
