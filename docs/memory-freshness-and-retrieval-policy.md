## 正式问题定义

在异步更新的长期记忆系统中,源信息已经发生变化时,记忆写入、索引和召回可能
仍停留在旧状态。旧记忆一旦以可信事实的形式进入模型上下文,会形成比普通生成
幻觉更隐蔽、更持久的错误:后续回答可以语言流畅、推理自洽,但事实基础已经失效。

系统同时面临召回策略两难:

- 不主动召回记忆,会在真正需要历史上下文时漏掉关键事实;
- 每轮对话强制召回,会引入无关或过期信息,增加 token、延迟和干扰;
- 仅按语义相关性召回,无法判断命中内容是当前状态、历史状态还是已经失效的状态。

需要解决的不是简单的“是否启用记忆”,而是:

> 如何在异步更新条件下识别并优先使用最新有效事实,保留但正确标记历史状态,
> 阻止过期记忆冒充当前事实,并根据本轮请求动态决定是否召回以及召回哪些记忆。

## 已识别的失败模式

1. **旧状态覆盖新状态**:设计文档或事实源已经更新,召回结果仍是旧版本。
2. **历史事实冒充当前事实**:旧记忆本应保留用于历史查询,却被当前状态查询直接采用。
3. **异步一致性窗口不可见**:写入已发生,抽取、索引或冲突处理尚未完成,调用方不知道
   当前读取是否可能陈旧。
4. **记忆幻觉缺少来源约束**:抽取事实没有足够精确的来源、时间或版本信息,模型无法
   核验其有效性。
5. **该召回时没有召回**:依赖宿主或模型偶然发起记忆搜索,导致必要上下文缺失。
6. **不该召回时强制召回**:无记忆需求的请求也注入历史内容,制造噪声、延迟和成本。
7. **冲突处理破坏历史查询**:简单删除或全局降权旧事实,可能让“过去是什么”无法回答。

## 后续能力目标

后续 feature 至少需要定义并验证以下能力,但本文不预设具体算法:

- 区分 `mention_time`、`event_time`、写入时间、索引可见时间和失效/替代时间;
- 显式表达事实之间的更新、补充、冲突和历史版本关系;
- 当前状态查询优先最新有效事实,历史查询仍可恢复对应时间的旧事实;
- 事实进入 reader 前携带可核验的来源指针,无来源的 claim 不获得事实权威;
- 对异步写入提供可观测的一致性状态,明确何时满足 read-your-writes;
- 使用确定性的高置信线索和安全默认值判断本轮是否需要记忆;
- 在召回关闭、信号缺失或状态不确定时诚实降级,不把猜测包装成记忆;
- 对延迟、token、陈旧命中率、错误状态采用率和非必要召回率建立评测门禁。

## 验收方向

正式立项时应把下列场景变成可重复测试和评测集:

1. 同一实体与 slot 先后出现不同值,当前查询不得采用已被替代的旧值。
2. 带明确历史时间的查询必须能够使用当时有效、现在已经过期的值。
3. 写入完成但异步索引未完成时,接口必须暴露状态或提供一致性读取路径。
4. 不依赖历史信息的请求不得无条件注入整批记忆。
5. 需要历史信息的请求不得因为模型没有主动调用搜索而静默漏召回。
6. 每个作为事实使用的结果都能回溯到原始 source span 及其时间信息。
7. 在相同 reader 和 token 预算下,新机制降低陈旧事实采用率,且不显著伤害历史查询。

## 与现有方向的关系

- [`memory-strategy.md`](./memory-strategy.md) 记录产品与论文战略;本文补充一个尚未
  解决的真实用户问题,不改变既有战略结论。
- PCIC 关注固定候选和预算下的干扰控制,可以抑制错误 slot、重复项并保护互补证据;
  但它不能单独解决异步可见性和 read-your-writes。
- provenance、时间状态建模和冲突关系是解决记忆陈旧与记忆幻觉的底座;
  动态召回策略解决“每轮都拉”和“完全靠模型主动拉”之间的产品问题。
- 当前项目已有 `event_start` / `event_end`、`superseded_by` 和 session 级来源字段,
  但尚未形成上述完整语义、精确 source-span 闭包或一致性契约。

## 立项约束

- 在 feature 完成前,文档和产品不得宣称 engram 已解决记忆新鲜度或异步一致性。
- 实现必须保留历史证据,不能以删除旧原文换取表面上的当前状态正确率。
- freshness、召回门控、reader prompt 和回答模型的变化必须分别消融,避免归因混杂。
- 任何触及写入、检索、冲突处理或存储的实现都受 LoCoMo/LongMemEval 可比评测门禁
  约束,并需增加专门的状态更新与异步一致性压力集。
